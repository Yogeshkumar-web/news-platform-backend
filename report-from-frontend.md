# Authentication & Authorization Report for Backend Team

> [!IMPORTANT]
> This report has been prepared for the Backend Team to understand how the Next.js Frontend manages Authentication and Authorization. Ensure that backend APIs align with the architecture below.

## Frontend Architecture Context
The frontend is built using **Next.js (App Router)**. It leverages a combination of Server Actions, Server Components, Client Components (`useAuth` via React Query), and Next.js Edge Middleware. 
There were previously multiple auth bugs in the frontend related to client-side Axios calls blocking cross-origin cookies. We have fixed these by implementing an **Edge Middleware API Proxy**.

---

## 1. How Tokens Are Handled on Frontend

The frontend uses an `httpOnly: true` cookie named `token` generated by Next.js to securely store the user's session token.

### Login Flow (`/api/auth/login`)
When a user logs in, Next.js executes a Server Action:
1. It sends a `POST` request to the Backend's `/api/auth/login` endpoint.
2. The Action currently extracts the token from the backend’s `Set-Cookie` header via a regex `match(/token=([^;]+)/)`.
3. It then sets its own `httpOnly: true` Next.js cookie (`sameSite: 'lax'`, `path: '/'`).

> [!TIP]
> **Backend Recommendation:** Ensure that the login endpoint also returns the token in the JSON response body (e.g., `{ success: true, data: { token: "ey..." } }`). This allows the frontend to parse the token safely without relying on brittle `Set-Cookie` string manipulation.

### Logout Flow (`/api/auth/logout`)
1. The frontend invokes a Server Action which calls a `POST` to the backend `/api/auth/logout`.
2. Regardless of the backend response, the frontend permanently deletes its Next.js `token` cookie to clear the session immediately.

## 2. API Request Flows (Important for Backend Auth Headers)

The backend team needs to know exactly how frontend requests are authenticated to write proper Authorization validation logic.

### Server-Side API Calls
Server Components use a utility called `serverFetch()` (`src/lib/api/server.ts`).
When `serverFetch()` calls the backend, it reads the Next.js cookie and explicitly attaches both headers:
- `Authorization: Bearer <token>`
- `Cookie: token=<token>`

### Client-Side API Calls
Client components use Axios (`src/lib/api/client.ts`).
**Crucial Change Made:** Axios now sends requests to a Next.js Edge proxy (`/api/proxy/*`) rather than calling the backend domain directly. 
When the Proxy intercepts the request, it extracts the Next.js `token` cookie and **automatically injects the `Authorization: Bearer <token>` header** before forwarding the request to the Backend.

> [!NOTE]
> **What This Means For Backend:** Every single authenticated request originating from this frontend—whether from Server Components or Client Components—will contain the **`Authorization: Bearer <token>`** header!

## 3. Backend Team Action Items

To fix the Authentication & Authorization logic on the backend, please review and implement the following adjustments:

1. **Standardize Auth Header Reading** 
   Since all frontend requests now inject `Authorization: Bearer <token>`, the backend **MUST** validate authorization primarily using the `Authorization: Bearer` header. Relying solely on `req.cookies.token` may cause mismatches in certain environments.
   
2. **CORS is No Longer an Issue** 
   Because the frontend utilizes an Edge API Proxy (`/api/proxy`), all browser AJAX requests act as same-origin requests to Next.js. You don't have to troubleshoot `Access-Control-Allow-Credentials` or cookie cross-site (`SameSite=lax`) issues with the Frontend anymore.
   
3. **Add `/api/auth/me` Endpoint**
   The frontend session validation (`getSession()` in `session.ts` and `useAuth()` hook) highly depends on a GET request to `/api/auth/me`. Make sure this endpoint strictly validates the Bearer token and returns the current User object.
   
4. **Role-Based Access Control (RBAC)**
   The frontend restricts pages based on roles matching specific strings (`ADMIN`, `SUPERADMIN`, `WRITER`, `SUBSCRIBER`). Make sure the token payload or the `/api/auth/me` user object contains the precise, exact case-sensitive `role` property.

5. **Token Secrets & Verification Phase**
   Ensure the JWT secrets/keys matched during the token generation process are identical across all deployed backend environments.
